## 1. Goals & non-goals

### Goals (v1.1.0 delivers these)

- **Brand rename.** The project moves under the new GitHub org `chamber-19` and
  adopts the package name `desktop-toolkit` across all layers (JS npm package,
  Python distribution, directory names, all internal imports).
- **Proper npm publishing.** Publish `@chamber-19/desktop-toolkit` to the
  GitHub Packages npm registry on every `v*` tag push. Consumers `npm install`
  from the registry instead of using `git+https://` subpath URLs.
- **"Fresh consumer install" CI job.** An automated check that actually
  installs the published package into a clean project and verifies every
  declared export resolves. This is the test that would have caught the v1.0.1
  git-subpath bug.
- **Consumer documentation.** Clear `.npmrc` + `package.json` setup instructions
  so any new consumer can onboard without guesswork.
- **All CI updated** to use new directory names and module names.

### Non-goals (explicitly out of scope for v1.1.0)

- Updating Transmittal-Builder to consume the renamed package ‚Äî that is Phase 4
  and gets its own planning doc.
- Tagging Transmittal-Builder v6.0.0.
- PyPI publishing for the Python package ‚Äî `pip`'s `git+https://` installs work
  fine; no equivalent of the npm subpath bug exists for Python. Revisit if
  friction emerges.
- Any new framework features (no new utilities, no new template components).
- Drawing-List-Manager integration.
- Codespaces or GitHub Advanced Security configuration for the new org.

## 2. Final naming decisions

These decisions are **locked**. The plan captures them; it does not re-litigate them.

| Layer | v1.0.x (current) | v1.1.0 (target) |
|---|---|---|
| GitHub repo | `Koraji95-coder/kc-framework` | `chamber-19/desktop-toolkit` |
| npm package | `@koraji95-coder/kc-framework` | `@chamber-19/desktop-toolkit` |
| JS package directory | `js/packages/kc-framework/` | `js/packages/desktop-toolkit/` |
| Python distribution name | `kc-framework` | `chamber19-desktop-toolkit` |
| Python import name | `kc_framework` | `chamber19_desktop_toolkit` |
| Python package directory | `python/kc_framework/` | `python/chamber19_desktop_toolkit/` |

### Rationale

**Why `desktop-toolkit` instead of `framework`?**
"Framework" implies an opinionated programming model that constrains how you
write code. What this package actually delivers is a small, focused collection
of composable utilities (PDF merge, email sender, splash screen, backend URL
resolver, update window). "Toolkit" is a more honest description of that
contract. It also reads better as a product name: "desktop-toolkit" implies
you grab what you need from a toolbox, which matches the actual consumption
pattern.

**Why `chamber-19` for the org?**
The original name `room-19` was already taken on GitHub (claimed since 2020).
`chamber-19` preserves the intended vibe ‚Äî a sealed, specific, slightly
ominous workshop space ‚Äî while being available. Both the GitHub org and
`@chamber-19` npm scope were confirmed free and have been claimed.

**Why the long-form Python import name `chamber19_desktop_toolkit`?**
Python convention is underscores for import names. Using the full
`chamber19_desktop_toolkit` (rather than, say, `desktop_toolkit`) ensures
global uniqueness if the package is ever published to PyPI. `desktop_toolkit`
is almost certainly taken. The long form future-proofs PyPI publishing without
requiring a later disruptive rename.

**Why not publish to PyPI in v1.1.0?**
`pip`'s `git+https://` installs work correctly with the `#subdirectory=python`
fragment. There is no equivalent of the npm-package-arg/pacote subpath bug for
Python. Publishing to PyPI adds complexity (account, token management, release
automation) that isn't justified yet. The Python package still gets renamed for
consistency; it just doesn't get published to a registry until a genuine need
arises.

## 3. Background ‚Äî why we're publishing instead of using git installs

> **TL;DR:** v1.0.1 `git+https://` npm installs are irretrievably broken in
> npm 11.x. Two separate root causes were confirmed. Publishing to GitHub
> Packages makes the problem disappear entirely.

The Transmittal-Builder PR #63 agent investigation confirmed two distinct
failure modes:

1. **`&path:` fragment ignored by npm-package-arg.**
   The v1.0.0 / v1.0.1 install syntax
   `git+https://github.com/Koraji95-coder/kc-framework.git#v1.0.1&path:js/packages/kc-framework`
   results in npm-package-arg silently discarding the `&path:js/packages/kc-framework`
   portion. npm resolves the ref to `v1.0.1` but clones the entire monorepo
   root into `node_modules/`, not the JS package subdirectory.

2. **`::path:` syntax parsed but not honoured by pacote.**
   The alternative `#v1.0.1::path:js/packages/kc-framework` syntax is
   understood by npm-package-arg (it parses `gitCommittish: v1.0.1` and
   `gitSubdir: /js/packages/kc-framework`), but pacote's extraction step
   still copies the monorepo root into `node_modules/`. The `${tmp}${gitSubdir}`
   path substitution in pacote's `git.js` handler appears not to be honoured
   during the actual file copy.

**Decision:** stop chasing this. Both failure modes are inside npm's internals.
Fixing them requires either patching npm/pacote (not our codebase) or switching
to a proper registry publish. We switch to a registry publish. The entire class
of problem disappears.

### 5.1 The publish-before-or-after-transfer question (critical ordering constraint)

**Open question:** Can the GitHub Actions workflow publish `@chamber-19/desktop-toolkit`
to GitHub Packages from a repository that is still owned by `Koraji95-coder`?

**Research / answer:**

GitHub Packages npm registry enforces that the npm scope matches the repository
owner. The `GITHUB_TOKEN` generated for a workflow run in
`Koraji95-coder/kc-framework` is scoped to the `Koraji95-coder` owner. That
token carries `packages: write` only for packages owned by `Koraji95-coder`, so
it **cannot** publish to the `@chamber-19` scope.

There are two workarounds:

**Option A ‚Äî Use a PAT (Personal Access Token) stored as a repo secret.**
A classic PAT issued by the `chamber-19` org admin (or a user who is an admin
of that org) with the `write:packages` scope can publish to `@chamber-19`
regardless of which repo the workflow runs in. Steps:
1. Create a classic PAT with `write:packages` on an account that has org-admin
   rights in `chamber-19`.
2. Store it as a secret `CH19_PACKAGES_TOKEN` in the current `Koraji95-coder/kc-framework` repo.
3. Use it in the publish workflow instead of `GITHUB_TOKEN`.

This works but is operationally messier (PAT rotation, secret in the "old" repo).

**Option B ‚Äî Reorder phases: do Phase 3 (transfer) before Phase 2 (publish).**
Once the repo lives at `chamber-19/desktop-toolkit`, the auto-generated
`GITHUB_TOKEN` naturally has `packages: write` for `@chamber-19`. No PAT
needed. The publish workflow is dead-simple.

**Recommendation: adopt Option B.** Reorder the execution to:

```
Phase 1 ‚Üí Phase 3 (transfer + rename) ‚Üí Phase 2 (publish + v1.1.0 tag)
```

This has zero downside. The renaming work in Phase 1 is identical regardless of
order. The transfer (Phase 3) is a few UI clicks that take minutes. Only after
the transfer does the publish workflow get added, and then it uses the native
`GITHUB_TOKEN`. No long-lived PAT, no secret management, no expiry risk.

The remainder of Phase 2 assumes the reordered sequence (repo already transferred
when publish is added).

### 5.3 Publish workflow ‚Äî `.github/workflows/publish.yml`

```yaml
name: Publish

on:
  push:
    tags:
      - "v*"

permissions:
  contents: read
  packages: write

jobs:
  publish-npm:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          registry-url: "https://npm.pkg.github.com"
          scope: "@chamber-19"

      - name: Install dependencies
        run: |
          cd js/packages/desktop-toolkit
          npm install --ignore-scripts

      - name: Publish @chamber-19/desktop-toolkit
        run: |
          cd js/packages/desktop-toolkit
          npm publish --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Notes on this workflow:**

- `registry-url` on `setup-node` writes a project-level `.npmrc` pointing at
  GitHub Packages and wires `NODE_AUTH_TOKEN` automatically.
- `scope: "@chamber-19"` ensures the generated `.npmrc` line is
  `//npm.pkg.github.com/:_authToken=‚Ä¶` scoped to the right registry.
- `--access public` is required; GitHub Packages treats scoped packages as
  private by default and will 402 without this flag.
- `--ignore-scripts` on install is a safety measure in CI; the package has no
  `prepare` or `postinstall` scripts so it has no effect in practice.
- The `permissions` block at the job level (`packages: write`) is what allows
  the `GITHUB_TOKEN` to push to GitHub Packages. This only works when the repo
  is owned by `chamber-19` (see ¬ß5.1 ordering discussion).

### 5.4 Fresh-consumer install CI job

Add the following job to `.github/workflows/ci.yml`. This job proves the
published package actually installs correctly into a clean downstream project ‚Äî
the test that would have caught the v1.0.1 git-subpath bug.

```yaml
  fresh-consumer-install:
    runs-on: ubuntu-latest
    needs: []          # no upstream dependency; runs in parallel with other jobs when triggered on a tag push
    if: github.ref_type == 'tag' && startsWith(github.ref, 'refs/tags/v')
    permissions:
      contents: read
      packages: read
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          registry-url: "https://npm.pkg.github.com"
          scope: "@chamber-19"

      - name: Create temp consumer project
        run: |
          TMPDIR=$(mktemp -d)
          echo "CONSUMER_DIR=$TMPDIR" >> "$GITHUB_ENV"

          # Minimal package.json (no deps yet)
          cat > "$TMPDIR/package.json" <<'JSON'
          {
            "name": "consumer-smoke-test",
            "version": "0.0.1",
            "private": true,
            "type": "module"
          }
          JSON

          # .npmrc pointing at GitHub Packages for @chamber-19 scope
          cat > "$TMPDIR/.npmrc" <<'NPMRC'
          @chamber-19:registry=https://npm.pkg.github.com
          //npm.pkg.github.com/:_authToken=${NODE_AUTH_TOKEN}
          NPMRC

      - name: Install published package into temp project
        run: |
          cd "$CONSUMER_DIR"
          npm install @chamber-19/desktop-toolkit
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Verify all declared exports resolve
        run: |
          cd "$CONSUMER_DIR"
          node --input-type=module <<'JS'
          import { createRequire } from "node:module";
          import { resolve } from "node:path";
          import { existsSync } from "node:fs";
          import { readFileSync } from "node:fs";

          const pkgPath = resolve("node_modules/@chamber-19/desktop-toolkit/package.json");
          const pkg = JSON.parse(readFileSync(pkgPath, "utf8"));

          let failed = false;
          for (const [subpath, target] of Object.entries(pkg.exports || {})) {
            const t = typeof target === "string" ? target : (target.default || target.import);
            if (!t) continue;
            const abs = resolve("node_modules/@chamber-19/desktop-toolkit", t);
            if (!existsSync(abs)) {
              console.error("MISSING:", subpath, "->", t);
              failed = true;
            } else {
              console.log("OK:", subpath, "->", t);
            }
          }
          if (failed) process.exit(1);
          JS
```

**Why this job matters:**

The existing `js` CI job validates that the source tree's exports map points at
files that exist *in the repo*. The `fresh-consumer-install` job validates that
the *installed npm package* ships those same files. These are different checks.
If `npm publish` accidentally publishes the monorepo root (the exact scenario
that bit v1.0.1 consumers), the fresh-install job catches it before anyone pins
the broken version.
